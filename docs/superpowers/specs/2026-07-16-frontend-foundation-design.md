The per-task review pipeline caught and fixed real issues beyond the plan on nearly every task: a critical unauthenticated RCE in the originally-scaffolded Next.js version (CVSS 10.0, patched before the first commit); caret-pinned Tailwind devDependencies exact-pinned to match the codebase's own convention (Task 2); API-client hardening for validation errors, import boundaries, and param safety (Task 5); a network-failure gap in the proactive refresh timer (Task 6); missing `Cache-Control: no-store` propagation on login/refresh responses (Task 7); a redundant prefix check in middleware (Task 8); `LoginForm` hardened against network failure, double-submit, and a dead-end TOTP step (Task 9); `RegisterForm`'s register-succeeds-login-fails path made recoverable instead of silently discarding its own error message (Task 10); session hydration from the refresh cookie restored on mount, with company-fetch HTTP errors surfaced instead of swallowed (Task 11); `MfaPanel`'s idle-state branches gated on a confirmed session rather than just hydration completion (Task 12); and E2E test isolation, selector, and timeout-scoping tightened (Task 13). Known-and-deliberately-not-fixed, all surfaced during Tasks 7, 10, 12, and 13's reviews and recorded inline in the plan at the time: (1) **multi-tab refresh race** — `AuthContext` gives each browser tab its own independent refresh timer with no cross-tab coordination; two tabs with closely-aligned refresh schedules can trigger the backend's single-use rotation plus family-level reuse detection, eventually logging both tabs out (the losing tab immediately, the winning tab silently on its next cycle) — the backend behaving correctly against an unsynchronized caller, not a backend defect; a `BroadcastChannel`-based single-leader refresh is the fast-follow candidate. (2) **Register email enumeration** — `POST /auth/register`'s distinct `409 "Email already registered"` lets anyone enumerate registered emails for free, at odds with the backend's own constant-time login path; the real fix touches `backend/app/routers/auth.py`, out of this worktree's scope. (3) **`clearSession()` in-flight-refresh race** — `clearSession()`'s `clearTimeout` only prevents a not-yet-fired scheduled refresh; an already-in-flight refresh fetch can still resolve afterward and silently reinstate a stale session, a narrow (~14 min interval) window sharing its root cause with (1) and (4). (4) **Strict-Mode double-refresh in dev** — `next.config.ts` has no `reactStrictMode` override (defaults `true`), and Strict Mode's dev-only double-invoke-effects behavior means `AuthContext`'s mount-hydration effect fires two real `/api/auth/refresh` network calls (only the second's result is ever applied to state, thanks to the effect's `cancelled` guard), reachable within a single tab, dev-server-only, same root-cause family as (1) and (3). (5) **Company-switcher backend gap** (Decision 5) — confirmed still open: the frontend deliberately ships read-only company display because no backend route exists to list a user's companies. Items (1), (3), and (4) share a root cause and likely a shared fix — a monotonic "session generation" counter in `AuthContext` that async refresh completions check before calling `setState` — noted here as a natural single fast-follow rather than three separate ones.

## Why now, and why decomposed this way

Five backend phases (auth, CRM, Project Management, Estimation, Compliance, Billing, Invoicing, Integrations) plus tier gating and full auth hardening are merged to `main` with zero consumer-facing UI — the product cannot currently be used by a human. Building screens for every phase at once is exactly the kind of multi-subsystem project the brainstorming process exists to decompose, so this spec covers **only** the prerequisite layer every later screen depends on: authentication, session handling, tenant switching, the generated API client, and the design-system scaffolding. User-confirmed decomposition (2026-07-16): Foundation first, then CRM+PM, Estimation+E-Signature, Compliance+Billing, Invoicing+Reporting, Integrations+Admin — each its own spec → plan → build cycle, mirroring how the backend itself was phased.

## Decisions

1. **Route groups separate marketing from product, with zero marketing-site disruption.** `frontend/app/(marketing)/` holds the existing `page.tsx`, `about/`, `security/`, `solutions/`, `early-access/` exactly as they are today — Next.js route groups (`(name)`) don't affect the URL, so `/about` keeps working unchanged. A new `frontend/app/(app)/` group holds every product route with its own root layout (nav, sidebar, company switcher) — Tailwind and shadcn/ui apply only inside this group. The marketing group's `styles.css` and `components.tsx` are untouched.

2. **In-scope routes for Foundation, and nothing more:** `/login`, `/register`, `/dashboard` (an authenticated placeholder landing page — no real content; that's every later sub-project's job), `/account` (profile display + MFA enroll/activate/disable, since the backend for this is fully built and it's squarely session/auth territory, not a later sub-project's concern).

3. **Tailwind CSS + shadcn/ui**, matching docs/03's stack table exactly. shadcn/ui is copy-paste-owned (components land in the repo, not a black-box npm dependency) and built on accessible Radix primitives — the standard, low-friction pairing with Tailwind for an admin/dashboard product, and avoids picking a heavier, more opinionated component framework (MUI, Ant) that would fight Tailwind's utility-first model.

4. **Generated API types, never hand-written**, per docs/03's explicit rule. `npm run generate:api-types` runs `openapi-typescript` against the backend's `/openapi.json` (the backend already serves this — FastAPI generates it automatically) and writes `frontend/lib/api/types.ts`. This is a manual/CI step, not a build-time dependency on a live backend, so `next build` doesn't require a running API. A thin fetch wrapper (`frontend/lib/api/client.ts`) supplies the typed request/response shapes and centralizes the two headers docs/03 Section 5.3 requires: `Authorization: Bearer <jwt>` and `X-Tenant-ID` for the active branch.

5. **Backend-for-frontend session handling — the one real architectural decision this spec makes.** The backend's refresh token (docs/superpowers/specs/2026-07-16-auth-token-lifecycle-design.md) is a bearer secret with a 14-day lifetime; storing it anywhere JS can read (`localStorage`, a non-httpOnly cookie) is an XSS-exposed design a construction-financial-data product shouldn't accept. Instead:
   - Next.js **Route Handlers** (`app/(app)/api/auth/login/route.ts`, `.../refresh/route.ts`, `.../logout/route.ts`) are the *only* code that ever calls the FastAPI backend's `/auth/login`, `/auth/refresh`, `/auth/logout`. The browser never talks to the FastAPI backend's auth endpoints directly.
   - The refresh token is set as an **httpOnly, Secure, SameSite=Lax cookie** by the Route Handler's response — never exposed to browser JS.
   - The access token is returned to the client and held in memory only (a React context), for the tab's lifetime; a background timer refreshes it (via the `/refresh` Route Handler, which also rotates the httpOnly cookie per the backend's rotation-on-use design) shortly before its 15-minute expiry.
   - **Route protection**: Next.js `middleware.ts` checks for the presence of the session cookie on every `(app)/*` request and redirects to `/login` if absent. (Presence-only check — the cookie's *validity* is enforced by the backend on the next actual API call; middleware doesn't decode or trust the JWT itself, avoiding a second source of truth for token validity.)
   - **Login flow**: email + password submitted to the login Route Handler. If the backend responds `401 "TOTP code required"`, the form reveals a second-step TOTP input and resubmits with the code — mirroring the backend's own two-step design (password proven first, TOTP only requested after). A successful login carrying `mfa_enrollment_required: true` (the backend's admin nudge signal) redirects to `/account` instead of `/dashboard`, with a banner prompting enrollment.
   - **Company display, not yet a functional switcher**: the backend has no route to list a user's full set of company memberships today (confirmed — `backend/app/routers/companies.py` only has `GET /companies/{company_id}` and `POST /companies/{company_id}/children`, both single-company operations). Foundation's app shell shows the current company's name, read-only, fetched via `GET /companies/{id}` using the login response's `default_company_id`. A functional multi-company switcher (dropdown, `X-Tenant-ID` header wiring) is deferred until a "list my memberships" backend route exists — a small, new backend gap this spec surfaces but doesn't fix, since no current sub-project's UX depends on switching between branches yet. When that route is added, the `X-Tenant-ID` header plumbing described in Decision 4's API client wrapper is what will consume it.

6. **Testing: adopt Playwright for the first time in this codebase**, per docs/10's original (never-honored-by-the-backend) tool choice. One E2E spec: register a new company → land on `/dashboard` → log out → log back in → land on `/dashboard` again, run against a live backend (the existing Docker Compose stack). This is Foundation's actual proof of done — a real human authentication loop working end-to-end, not just component-level tests. Component/unit tests (Vitest + React Testing Library) are used opportunistically for pure logic (the TOTP two-step form state, the token-refresh timer) but are not a completeness gate the way the Playwright spec is.

7. **CI**: extends `.github/workflows/backend-ci.yml`'s pattern with a new `frontend-ci.yml` — `next build` (type-checks as a side effect of the TypeScript compile) + `next lint` (ESLint, Next.js's built-in config) on every push. The Playwright E2E spec runs against the Compose stack, matching how the backend's own manual E2E verification works today; wiring Playwright into CI itself (vs. a manual pre-merge step, matching the backend's current practice) is left as a follow-up, not blocking this spec — the backend suite doesn't run its E2E script in CI either, and Foundation shouldn't hold a higher bar for CI automation than the system it's extending.

## Explicitly out of scope (future sub-projects' territory)

- Any CRM, Project Management, Estimation, Compliance, Billing, Invoicing, or Integrations screen — those are sub-projects 2-6.
- `GET /companies/{id}/users` (a small missing backend route, unrelated to this work) and the never-published `PROJECT_COMPLETED` event — both pre-existing, tracked gaps this spec doesn't touch.
- Wiring Playwright into CI as an automated gate (see Decision 7).
- Real-time/websocket notification handling for the async job pattern docs/03 Section 6 describes (PDF export, sync jobs) — Foundation has no screens that trigger async jobs yet.
- Dark mode, i18n, or any visual polish beyond a functional, accessible baseline — this is infrastructure, not a design pass.
